Relay Code: manage AI coding agents from your phone.

AI coding agents finish their work even when you are not at your desk. Relay Code is a mobile app for guiding agents, reviewing their code, and shipping it. It is built for the six free minutes between meetings, not for a long desk session.

01 · Overview

An IDE needs a desk. An agent doesn't.

Relay Code connects to a developer's GitHub repositories and lets them run AI coding agents on real work: understand new code, find and debug problems, write fixes, run the test suite, open pull requests, and merge with human approval. The whole flow is built for mobile first.

Who it's for: developers who already use AI coding tools and found a new problem. The agent works on its own, but reviewing its work still needs a laptop. On-call engineers, team leads who check agent output, and founders who work between meetings.

The problem it solves: with coding agents, the slow part is no longer writing code. The slow part is waiting for a human to answer. Relay Code cuts that waiting time from "when I'm back at my desk" to "right now, from my phone."

02 · Problem

The agent finished 40 minutes ago. You're still on the train.

User pain

Developers who use coding agents told us the same story again and again. The agent finishes a task in 5 to 15 minutes, and then waits for review or for an answer to a question. Sometimes it waits for hours, because the developer is away from the desk. And when they tried to unblock the agent from a phone, GitHub mobile made it feel unsafe: the diffs are hard to read, there is no test information, and there is no way to give the agent new instructions without typing a long message.

Business pain

For teams that pay for agent platforms, a blocked agent is wasted money. Seats and compute sit idle while the queue waits. Data from our design partners showed that the average agent task spent 3.4 times longer waiting for a human than actually working. Time-to-merge, the number these teams care about, was mostly human waiting time.

03 · Goals

Make the phone a real place to make an engineering decision.

One sentence kept the team aligned for seven months: a developer should be able to take an agent's task from "needs review" to "merged" in under five minutes, with one hand, and without regret.

04 · Research

Fourteen developers, three weeks, one surprise.

Interviews with 14 developers who already use coding agents (5 senior engineers, 4 team leads, 3 founders, 2 on-call SREs), a one-week diary study of agent usage, and a review of 6 similar products.

We expected the hard problem to be input: how do you type instructions on a phone? It wasn't. Voice and short corrections solved input easily. The hard problem was trust: what does a developer need to see, on a small screen, to approve code they didn't write?

Trust comes from evidence, not from chat

Every participant asked for the same three things: what changed (the diff), does it work (the tests), and what was the plan (the agent's log). Nobody trusted chat messages alone. "The agent saying 'done' means nothing."

Mobile sessions are 2 to 6 minute bursts

The diary study showed that developers check their phones in short windows: between meetings, on the way, in lines. Any flow longer than about 5 minutes was dropped and pushed to the desk. That kills the whole point of the product.

Fear of the wrong tap is real

Participants talked about the fear of merging by mistake. If a dangerous action is one accidental tap away, the whole screen feels unsafe, and people avoid mobile review completely.

Short corrections beat long instructions

Nobody wanted to write detailed prompts on a phone. They wanted to say small things: "use the existing helper", "add a test for the empty case", "wrong file". Corrections are short. Design for corrections.

Test results create confidence

A green test suite moved people from "I'll wait for my desk" to "I'll merge from my phone" more than anything else. But only when they could see which tests ran, not just a checkmark.

"I don't need to read every line on my phone. I need to know what changed, that the tests prove it, and that I can take control if it's wrong." P7, senior engineer, interview round 2

07 · Information architecture

Sessions, not repositories.

The key IA decision: the main object is the session (one agent, one task), not the repo. On mobile, developers think "what needs me?", never "let me browse a repo." Repos exist, but one level down.

Hierarchy rule: anything that needs a decision lives at most 2 taps from launch. Browsing and settings can live deeper. The Inbox is the default tab because research showed that 80% of sessions start from a notification. The app opens on the answer to "why am I here?"

08 · User flow

The five-minute loop.

The core flow, from notification to merged, with the two decision points that define the product. We asked about every box: can this step be skipped, delayed, or merged with another?

Decisions I would defend:

  • "Move to desk" is a planned outcome, not a failure. Research showed that forcing every review to finish on mobile kills trust. Saving a session for the desk keeps the phone honest, and keeps users coming back.
  • Steering goes back to review, not to an empty chat. A correction re-runs the agent and creates a fresh diff. The user never leaves the evidence.
  • Tests control the merge button. The merge control does not appear at all when tests are red or stale. Safety built into the layout works better than warning dialogs.
09 · Wireframes

Three rounds, one big change.

Low-fi tested the reading order of the review screen. Mid-fi fixed layout and thumb reach. The big change between rounds: from a chat-based screen to an evidence-based screen.

Lo-fi · Round 1: chat first

Review as a chat with a diff card inside. Testers scrolled past the code and read the agent's words instead. Dropped: words won over evidence.

Lo-fi · Round 2: evidence first

Plan summary, then diff, then tests. Chat moved behind a tab. Time to decision in tests dropped about 40%. Kept.

Mid-fi · Round 3: hunk cards + thumb bar

The diff is split into swipeable hunk cards. All decisions moved to a fixed bottom bar, inside thumb reach. Tested one-handed on 5 device sizes.

What we learned between rounds. Round 1 failed in a useful way. When the agent's story comes first, users skim the code instead of reading it. That is the opposite of trust. Round 2 flipped the order. Round 3 fixed the ergonomics: in usability tests, every button at the top of the screen was missed or needed two hands. The fixed bottom bar solved reach and gave the merge control one stable place.

10 · Design exploration

Three visual directions, one winner.

All three were built as coded prototypes, not static images, and tested with 8 users on their own phones, outside and on the move, because that is where the product lives.

A: "Terminal"

Dark only, heavy monospace, log-line look. Spoke the developer's language and felt natural for CLI users.

Rejected. Dense mono text failed badly in sunlight, and it made Relay feel like a monitoring tool, not a decision tool.

B: "Messenger"

Chat-based, the agent as a contact. Zero learning curve and the friendliest option for non-experts.

Rejected. The chat idea worked against the main job: bubbles hide the code, and approving code inside a chat felt like agreeing to a text message.
Shipped

C: "Mission control"

A calm card system: each session is a card with a status, the evidence stacked inside, and decisions on a fixed bar. Light and dark. High contrast, big targets.

Selected. Fastest time to decision in testing, easiest to read outside, and the same visual language worked on phone, tablet and web.

The deciding test was simple and cheap: give a user a phone at a bus stop, with a real pending review, and time them until a confident decision. Concept C won by 35 seconds on average. But the deeper reason we chose it was strategic: cards carry status, and status is the product. You cannot scan a chat. You cannot sort a terminal. You can do both with a stack of clear cards.

14 · Challenges

The four hardest decisions.

14.1Speed vs. safety at the moment of merge

Considered: one-tap approve
The fastest possible loop, and on paper it matched the five-minute goal. But usability tests showed it turned worry into avoidance: users were afraid to open reviews at all, because one wrong tap could ship code.
Shipped: hold-to-merge, only when tests are green
An 800ms press with a visible progress fill, shown only when tests are green on the current diff. It adds less than a second, and it removed the fear completely. After launch, the revert rate on mobile came in below desktop.

14.2Making diffs readable on a small screen

Considered: side-by-side diff, made smaller
Familiar from desktop, and power users asked for it. But at phone width each column got about 40 characters. Horizontal scrolling broke the reading flow and one-handed use.
Shipped: unified hunk cards with word-level highlights
Each hunk is a card with its own file header. Only the changed words get background color, so lines stay easy to scan. Short AI summaries per hunk give a reading order. Very large diffs go through "move to desk".

14.3Live agent output vs. battery and attention

Considered: a streaming terminal view
Fun to watch ("I can see it think") and impressive in demos. But it drained battery through open connections, and it taught users to sit and watch, which is the opposite of what the product promises.
Shipped: a checkpoint model
Agents report at meaningful points: plan ready, diff ready, tests done, question raised. Push notifications carry the checkpoint data, so the lock screen already answers "do I need to act?" Battery cost is near zero, and trust stayed the same.

14.4Technical limits: GitHub API rate limits and weak mobile networks

Considered: fetch everything live
Always fresh data and the simplest architecture. But on a subway connection, the review screen showed loading skeletons for 8 to 12 seconds. That is longer than many full mobile sessions.
Shipped: session snapshots plus queued actions
The backend prepares a complete review bundle (plan, diff, test results) at each checkpoint and pushes it. The phone opens instantly from cache and updates in the background. Approvals queue offline and confirm with a clear "merged" receipt, never a silent guess.

User impact

The change in feeling mattered as much as the numbers. In follow-up interviews, the phrase that came back again and again was "no more guilt." On-call engineers stopped taking laptops to dinner. Leads like Daniel moved from reviewing everything at 9pm to clearing the queue during the day. Activation reached 54% (first mobile approval within 48 hours).

Lessons learned

  • Trust is a layout problem. The single biggest improvement in the project was changing the order of the review screen: evidence before the agent's story. No model upgrade moved the numbers as much.
  • Design the "no" path with the same care as the "yes" path. "Move to desk" looked like giving up. It turned out to be the feature that made people trust the "yes".
  • Prototyping in code was not optional. Every important finding, thumb reach, sunlight reading, offline behavior, was invisible in static mockups and obvious on a real device.

What's next

  • Voice steering. Spoken corrections were the most requested feature in exit interviews.
  • Risk-based review depth: the system suggests "skim / review / desk" per session, and the user can always override.
  • Team spaces: shared inboxes and approval rules for companies running many agents.
