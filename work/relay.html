Relay: fix bugs from your phone.

Relay is a mobile-first AI engineering command center. It connects a GitHub repo to an AI agent with real codebase access, a file browser, a cloud terminal, and one-tap skills, in a single phone-native experience. I designed it and built it, end to end.

01 · Overview

Serious engineering work, on a 390px screen.

An AI agent with real codebase access, a file browser, a cloud terminal, and one-tap skills. For developers away from their desk but not away from their work.

02 · Problem

Developer tools assume you're at a desk.

Away from your machine, you can read a bug report but you can't act on it. AI agents made the work phone-sized: directing and checking, not typing. No tool offered that in a phone-native way.

Existing approachRepo accessAI agentTerminalPhone-nativeWhere it breaks
GitHub MobileRead + reviewNoneNoneYesYou can look, but you can't act. No agent, no shell, no fixes.
ChatGPT / Claude appsNoneChat onlyNoneYesThe AI can't see your code. You paste snippets and lose context.
SSH clients (Termius etc.)FullNoneYesTerminal onlyA raw shell on a phone keyboard. Powerful and painful.
Cloud IDEs on mobileFullSomeYesNoA desktop layout squeezed into a phone. Unusable with one hand.
03 · Goals

Mobile-first, without dumbing it down.

One line: "Fix bugs from your phone."

PrincipleHow it shows up in the product
Mobile-first, desktop-capableBottom nav, safe-area padding, touch targets, bottom sheets. A full sidebar layout on desktop, from the same components.
Context stays visibleActive project, connected repo, and branch are always shown, in chat and in file views. You never wonder "what am I working on?"
Progressive disclosureHub shows quick actions; chat reveals skills; the file view reveals line selection and Ask AI. Depth on demand, not up front.
Don't interrupt flowThe chat stays mounted when you switch to Terminal or Preview, so a streaming agent answer is never cut off.
Trust and clarityEmpty states say what to do next. Errors have retry actions. Destructive actions require confirmation.

Non-goals: not a code editor, not a Git client. A command center.

04 · Research

Living the problem, then studying it.

My own pain, competitor teardowns, and conversations with developers.

Phone sessions are short and interrupted

Minutes, not hours. Nothing important can be lost when you switch apps.

Context is the first thing you lose on mobile

Repo, branch, and project must always be visible, or users act on the wrong thing.

Typing is the enemy, choosing is fine

Everything important starts with a tap, not a paragraph.

Reading code on a phone is fine, if you design for it

Highlighting, line numbers, and tap-to-select make 390px enough.

05 · User personas

The two situations I designed for.

Less "invented people", more real situations that repeat. Every screen was checked against both.

B

The builder on the move

A developer between places, with an idea or a bug in mind
"I know exactly what's wrong. I just can't touch the code right now."
Situation
  • Commuting, waiting, or away for the evening
  • Has full context in their head, zero tools in their hand
Needs
  • Start an agent on the problem now, not in two hours
  • Check the agent's work: read the diff, run it, see the preview
  • One-handed use, short taps, no long typing
What Relay gives them
  • Hub → New Agent → quick agent presets (Debug, Review, Explore)
  • Chat with real codebase access + live terminal + preview
U

The unblocked teammate

Someone whose team is waiting on an answer from them
"They just need me to look at one file and say yes. And I'm in line at the supermarket."
Situation
  • A teammate is blocked on their review or their knowledge
  • The answer needs looking at real code, not memory
Needs
  • Open the exact file fast: search, breadcrumbs, highlighting
  • Ask the AI about the code they are looking at, in place
  • Answer in minutes and get back to their life
What Relay gives them
  • Code tab: repo search + file browser + line selection
  • Tap lines → Ask AI, with file path and line numbers pre-filled
06 · User journey

A bug report arrives at 8:40, on the train.

The same event, without Relay and with it. The journey that drove the whole design.

Stage
Alert
Understand
Act
Verify
Close
Without Relay
A crash report lands in Slack. You read it on your phone. That's all you can do.
You try to remember the code. GitHub mobile shows the file, tiny and read-only.
Nothing. You write "will look when I'm at my laptop" and feel behind all day.
Hours later you finally reproduce it at your desk. The context from the morning is gone.
The fix ships in the evening. Total time: a full day for a 20-minute bug.
With Relay
Same report. You open Relay. Your project and branch are already there.
You open the file in the Code tab, tap the suspicious lines, and hit Ask AI.
The agent explains the bug and proposes a fix. You watch it work, streaming, with tool calls visible.
You run the tests in the cloud terminal and open the live preview. It works.
Fix done before your stop. Total time: the length of a train ride.
Design takeaway
Open onto context, not onto a menu.
Code must be readable and selectable, not just viewable.
Acting = directing an agent. Make that one tap.
Terminal and preview must live next to the chat.
Session continuity: everything is saved, nothing is lost.

08 · User flow

From "something's broken" to a verified fix.

The core flow. The two loops in the middle (steer the agent, check the code) are where users actually spend their time.

  • Presets before prompts. The flow starts with a tap.
  • Verify without leaving. Terminal and preview are tabs; the chat stays mounted.
  • Sessions are permanent. Interruption is expected; loss is not allowed.

10 · Design exploration

Three looks for a tool you use in the dark.

A developer tool used on a phone, often at night, often outside. The visual language had to feel technical but stay calm.

A: "Friendly SaaS"

Light, blue, approachable. Code looks foreign on it, and it glows at night.

Rejected.

B: "Pure terminal"

Hacker black, mono everywhere. Honest, but tiring to read.

Rejected. Right for one tab, not the whole world.
Shipped

C: "Dark-first, emerald"

Deep dark base, soft cards, emerald accent. Mono only for code.

Selected. Calm at night, technical without being hostile.

14 · Challenges

The hardest problems weren't visual.

14.1Switching tabs without killing the agent stream

Separate pages
Navigating away unmounts the chat; a streaming answer is lost.
One shell, chat stays mounted
Streams keep flowing behind Terminal and Preview.

14.2Making code readable and selectable at 390px

Pinch-zoom a desktop view
More panning than reading; finger selection is hopeless.
Tap-to-select lines + Ask AI
Lines are the unit. Tap, then ask, with file and line context pre-filled.

14.3Keeping context visible on a small screen

Hide context to save space
A context bar costs a row of pixels.
The context bar never drops
A row of pixels is cheaper than a wrong action on the wrong repo.

14.4One form, two surfaces

Desktop modals everywhere
Centered modals fight the keyboard and ignore the thumb.
Dual-surface modals
One form, a BottomSheet on mobile and a modal on desktop.
