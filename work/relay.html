Relay: fix bugs from your phone.

Relay is a mobile-first AI engineering command center. It connects a GitHub repo to an AI agent with real codebase access, a file browser, a cloud terminal, and one-tap skills, in a single phone-native experience. I designed it and built it, end to end.

01 · Overview

Serious engineering work, on a 390px screen.

Relay connects to a GitHub repository and gives you everything you need to work on it from a phone: an AI agent that can actually read and change the codebase, a full file browser with syntax highlighting, a live cloud terminal, an app preview, and one-tap "skills" (preset agent workflows like Explore, Review, Debug).

Who it's for: developers who are away from their machine but not away from their work. Commuting, on a call, between meetings, or just on the couch.

The bet: serious engineering work can happen on mobile, if you design for thumbs, context, and interruption, instead of shrinking a desktop tool.

02 · Problem

Developer tools assume you're at a desk.

User pain

When you're away from your machine, you can't easily inspect code, run an agent, or unblock a teammate. A bug report arrives, a question comes up in review, an idea hits you on the train, and the only real answer is "when I'm back at my laptop". The work doesn't wait, but the tools do.

Why this got urgent now

AI agents changed the shape of the work. A lot of engineering is now directing and checking an agent, not typing every line. That kind of work fits a phone much better than writing code does. But no tool offered it in a phone-native way.

03 · Goals

Mobile-first, without dumbing it down.

The product vision in one line: "Fix bugs from your phone." These are the design principles I set at the start, and how each one shows up in the shipped product.

PrincipleHow it shows up in the product
Mobile-first, desktop-capableBottom nav, safe-area padding, touch targets, bottom sheets. A full sidebar layout on desktop, from the same components.
Context stays visibleActive project, connected repo, and branch are always shown, in chat and in file views. You never wonder "what am I working on?"
Progressive disclosureHub shows quick actions; chat reveals skills; the file view reveals line selection and Ask AI. Depth on demand, not up front.
Don't interrupt flowThe chat stays mounted when you switch to Terminal or Preview, so a streaming agent answer is never cut off.
Trust and clarityEmpty states say what to do next. Errors have retry actions. Destructive actions require confirmation.

Non-goals: Relay is not a code editor on a phone (typing large amounts of code on glass is a losing game), and not a Git client. It is a command center: you read, direct, check, and decide.

04 · Research

Living the problem, then studying it.

This started from my own pain as a developer who is often away from the desk. I backed it up with teardowns of GitHub Mobile, the ChatGPT and Claude apps, Termius, and Replit's mobile app, and with conversations with developer friends about when and how they use their phones for work.

Phone sessions are short and interrupted

Nobody sits down for an hour of phone work. Sessions are a few minutes, and they get cut by notifications, stops, and real life. The design must survive interruption: nothing important can be lost when you switch apps or tabs.

Context is the first thing you lose on mobile

On a desktop you have windows and tabs to remind you where you are. On a phone you have one small screen. If the repo, branch, and project are not always visible, users act on the wrong thing or stop acting at all.

Typing is the enemy, choosing is fine

Long prompts on a phone keyboard kill the flow. But tapping a preset, picking a suggestion chip, or selecting lines of code is easy. Everything important in Relay can start with a tap, not a paragraph.

Reading code on a phone is fine, if you design for it

Syntax highlighting, line numbers, horizontal code scroll inside a fixed frame, and tap-to-select lines make code readable at 390px. What breaks is desktop chrome around the code, not the code itself.

06 · User journey

A bug report arrives at 8:40, on the train.

The same event, without Relay and with it. The journey that drove the whole design.

Without Relay
A crash report lands in Slack. You read it on your phone. That's all you can do.
You try to remember the code. GitHub mobile shows the file, tiny and read-only.
Nothing. You write "will look when I'm at my laptop" and feel behind all day.
Hours later you finally reproduce it at your desk. The context from the morning is gone.
The fix ships in the evening. Total time: a full day for a 20-minute bug.
With Relay
Same report. You open Relay. Your project and branch are already there.
You open the file in the Code tab, tap the suspicious lines, and hit Ask AI.
The agent explains the bug and proposes a fix. You watch it work, streaming, with tool calls visible.
You run the tests in the cloud terminal and open the live preview. It works.
Fix done before your stop. Total time: the length of a train ride.
Design takeaway
Open onto context, not onto a menu.
Code must be readable and selectable, not just viewable.
Acting = directing an agent. Make that one tap.
Terminal and preview must live next to the chat.
Session continuity: everything is saved, nothing is lost.

08 · User flow

From "something's broken" to a verified fix.

The core flow. The two loops in the middle (steer the agent, check the code) are where users actually spend their time.

Decisions I would defend:

  • Presets before prompts. Hub quick agents map straight to agent prompts, so the flow starts with a tap. Writing a prompt is always possible, never required.
  • Verify without leaving. Terminal and Preview are tabs of the same workspace, and the chat stays mounted behind them. Checking the fix never costs you the conversation.
  • Sessions are permanent. After the first message the URL becomes /chat/{sessionId}. Interruption is expected; loss is not allowed.
09 · Wireframes

How the shell found its shape.

The main structural question: what is "home", and where do terminal and preview live? Three structures, tested as rough builds.

Study 1: chat is home

Open the app, land in a chat. Fast for one thing, blind for everything else: no way to see projects, files, or what you can do. Dropped: a chat box is not a command center.

Study 2: hub with quick agents

A dashboard: New Agent on top, active project card, a grid of preset workflows, recent sessions. Answers "what can I do right now?" at a glance. Kept.

Study 3: workspace shell with tabs

Chat, Terminal, and Preview as tabs of one shell, with the repo context bar on top. The key detail: switching tabs hides the chat but does not unmount it. Shipped.

What settled it. Study 1 failed the "cold open" test: a new user sees a text box and doesn't know what the product can do. Study 2 fixed discoverability, and Study 3 solved the deeper problem: verification (terminal, preview) and conversation (chat) are one workflow, so they must be one screen with tabs, not separate pages.

10 · Design exploration

Three looks for a tool you use in the dark.

A developer tool used on a phone, often at night, often outside. The visual language had to feel technical but stay calm.

A: "Friendly SaaS"

Light, blue, rounded, approachable. Nice for a landing page, wrong for the job: code and terminals look foreign on it, and it glows in your face at night.

Rejected. The product lives next to code; the surface should too.

B: "Pure terminal"

Full hacker black, mono everywhere. Honest about the domain, but tiring to read, and chat bubbles and cards fight the aesthetic.

Rejected. Too harsh as the whole world; right as one tab (the terminal itself).
Shipped

C: "Dark-first, emerald"

A deep dark base with soft cards, an emerald-to-cyan accent for brand moments, and monospace reserved for code and paths. Light mode exists as a token swap.

Selected. Calm at night, technical without being hostile, and code blocks feel native in it.

14 · Challenges

The hardest problems weren't visual.

14.1Switching tabs without killing the agent stream

The naive way: separate pages
Chat, terminal, and preview as separate routes. Clean code, terrible experience: navigating away unmounts the chat, and an in-flight streaming answer is lost. On mobile, where switching is constant, this happens all the time.
Shipped: one shell, chat stays mounted
The WorkspaceShell keeps the chat mounted and hidden when you switch to Terminal or Preview. Streams keep flowing in the background. This one decision is why the product survives real mobile usage.

14.2Making code readable and selectable at 390px

Considered: pinch-zoom a desktop code view
What every cloud IDE does on mobile. You spend more time panning than reading, and precise text selection with a finger is hopeless.
Shipped: tap-to-select lines + Ask AI
Lines are the selection unit, not characters. Tap lines to select them, then copy or send to the AI, with the file path and line numbers pre-filled. Selection became the main way people talk to the agent about code.

14.3Keeping context visible on a small screen

Considered: hide context to save space
Screen space is precious, and a context bar costs a row. The temptation is to drop it and trust the user's memory.
Shipped: the context bar is never dropped
Active project, connected repo, and branch are always shown in chat and file views. On a phone, memory is the scarcest resource. A row of pixels is cheaper than a wrong action on the wrong repo.

14.4One form, two surfaces

Considered: desktop modals everywhere
One modal component for all screens. On mobile, centered modals fight the keyboard, ignore the thumb, and feel like a website.
Shipped: dual-surface modals
The same form renders as a BottomSheet on mobile and a centered modal on desktop: connect repo, create project, Ask AI. One source of truth for the form, two native-feeling presentations.
15 · Results

Shipped, live, and used for real work.

Relay is in production on Vercel, used for real agent workflows on mobile and desktop.

What shipped

  • Full auth flow: register, login, OAuth, email verification
  • Hub dashboard with quick agents and project switching
  • Agent chat with streaming, visible tool calls, and skills integration
  • File browser with syntax highlighting, search, and tap-to-select → Ask AI
  • Cloud terminal (xterm.js) and live app preview in the workspace shell
  • Projects and repo management, with a PAT connect flow
  • Profile with stat cards, a 30-day activity heatmap, and agent memory topics
  • Settings: AI models and API keys, billing, repos, appearance, security, MCP tools
  • Landing page, features page, legal pages, light/dark theme

Lessons learned

  • The hardest UX problems weren't visual polish. They were keeping repo context visible, not breaking agent streams when switching tabs, and making code readable and selectable on a small screen.
  • Design for interruption first. Session URLs, mounted chat, saved history: every continuity feature earned its place the first time a train went into a tunnel.
  • Presets beat prompts on mobile. Quick agents and suggestion chips carry most sessions. Free-text prompts are the escape hatch, not the main path.

What's next

  • Voice input for steering the agent, hands-free
  • Push notifications when a long agent run finishes
  • Deeper PR flows: review and merge from the same shell
