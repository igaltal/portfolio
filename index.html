About

Designer and engineer, in one person.

I have worked on both sides: writing the tickets and closing them. I design with real production limits in mind, and I build with respect for the people who will use the product.

Many product decisions get lost on the way. A researcher learns something, a designer draws it, an engineer builds something a bit different, and the user gets the sum of all those small errors. I remove that chain. When I run the interviews, draw the flows, define the tokens, and write the components, the original idea is still there when the product ships.

I also work a lot with AI: I design AI products, and I use AI tools to build prototypes fast. But the rule I follow is simple: tools suggest, people decide. That rule shows up in my products too.

  • Problem before pixels

    I don't open a design tool before I can explain the user's problem, the business goal, and the number we want to move. A weak problem statement always leads to a weak interface.

  • Systems over screens

    One screen solves one case. A system solves all the next ones too. I start with tokens, components, and rules, so every new screen is faster to build and stays consistent.

  • Prototype in code

    Static mockups always look good. Working prototypes show the truth. I test with real data, real loading times, and real thumbs on real phones, because that is where designs fail.

  • Psychology matters

    Trust, fear, habit, mental load. These are design materials. Every button I ship is a decision about how a person will feel at that moment.

  • Business is a user too

    Good UX that doesn't help activation, retention, or revenue is a hobby. I define success metrics on day one and check them after launch.

Toolchain: Figma · design tokens · TypeScript · React / Next.js · SwiftUI prototyping · Framer Motion · Tailwind · Node · Anthropic & OpenAI APIs · prompt design · Git, CI, and all the boring parts of shipping.
